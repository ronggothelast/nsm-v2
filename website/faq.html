FAQ

Frequently asked questions

Common questions about Nift v2. Can't find your answer? Open an issue.

Why another static site generator?

Most SSGs (Hugo, 11ty, Jekyll) are written in scripting languages and optimize for developer convenience, not build speed. Nift v2 treats build time as a compiler problem: a typed AST, an incremental cache, a work-stealing executor, and SIMD scanning. The result is sub-millisecond rebuilds for most projects.

Nift v2 is also a single binary with zero runtime dependencies. No Node, no Ruby, no Python. Just nift build.

Why C++ and not Rust/Go/Zig?

The original Nift v1 was written in C++. The v2 rewrite continues that lineage because:

1. mmap + xsimd — direct access to SIMD intrinsics via xsimd, no FFI overhead.

2. sol2 for Lua — the most mature C++ ↔ Lua binding library.

3. Ecosystem — CMake, vcpkg, Catch2, cpp-httplib are all first-class C++ citizens.

4. Binary size — a stripped release binary is ~2-4 MB, smaller than most Go binaries.

How does Nift v2 compare to Hugo in speed?

Hugo (Go) is fast for a Go SSG. Nift v2 (C++) is faster for template scanning (2.4× with SIMD) and incremental rebuilds (BLAKE3 cache). On a 200-page site:

• Cold build: Nift ~95ms, Hugo ~180ms (varies by template complexity)

• Warm build (1 file changed): Nift ~6ms, Hugo ~40ms

Both are fast enough that the difference is imperceptible for small sites. The advantage grows with site size.

Is the template language backward compatible with v1?

Mostly yes. The core directives (@if, @for, @set, @include, @template) are unchanged. Minor differences:

• v1 @{name} and @{end} are auto-converted by the migrator to v2 @name{ } syntax.

• v1 $name variables work identically.

• v2 adds @def (reusable blocks), @write (file generation), and expression support in @set.

Run nift migrate on your v1 project to handle any syntax differences automatically.

How secure are Lua plugins?

Lua plugins run in a restricted sandbox:

• Removed: os.execute, io.open, require, loadfile, debug.*

• Memory cap: configurable, default 16 MB

• Instruction budget: prevents infinite loops

• No filesystem access: plugins can only read/write through the build context API

• No network access: no sockets, no HTTP calls from Lua

For untrusted plugins, you can further restrict with OS-level sandboxing (seccomp, AppArmor).

Does Nift v2 support Markdown?

Not directly in Phase 7. Markdown support is planned for Phase 8 via a built-in parser. Currently, you can:

• Write HTML directly in content files

• Use a Lua plugin to call an external Markdown processor

• Use a native C-ABI plugin that links a C Markdown library

Can I use Nift v2 with a CI/CD pipeline?

Yes. Nift v2 is a single binary with no runtime dependencies. Typical CI setup:

curl -LO .../nift-linux-x86_64.tar.gz && tar xzf ... && ./nift build

Exit code 0 = success. Non-zero = failure. The --quiet flag suppresses output for clean CI logs.

What about WASM support?

WASM (Emscripten) builds are planned for Phase 8-9. The goal is to run Nift v2 in the browser for client-side preview and in edge compute environments. The core crate has no platform-specific code, so porting is expected to be straightforward.

How do I contribute?

1. Fork the repo: github.com/ronggothelast/nsm-v2

2. Create a feature branch: git checkout -b feature/my-thing

3. Make changes, add tests (Catch2), run just test

4. Ensure clang-format passes: just fmt

5. Open a PR against main

See CONTRIBUTING.md for details.

What license is Nift v2?

MIT License. Use it for personal projects, commercial products, anything. No attribution required in output.

Where does the name "Nift" come from?

Nift was originally named "nift" (a play on "swift" without the 's' — reflecting its speed focus). The name stuck. The v2 rewrite preserves the name and the CLI binary is still called nift.
