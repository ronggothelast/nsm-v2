Plugin development

Nift v2 supports two plugin types: sandboxed Lua scripts for build-time logic, and native C-ABI plugins for performance-critical work.

Lua plugins

Lua plugins run inside a sandbox with a memory cap and instruction budget. They have access to the build context (pages, config, output paths) but cannot access the filesystem, network, or shell.

Example: sitemap generator

-- plugins/sitemap.lua
function on_build_complete(ctx)
  local pages = ctx.get_pages()
  local base = ctx.get_config("base_url")
  local xml = "<?xml version='1.0'?>\n<urlset xmlns='http://www.sitemaps.org/schemas/sitemap/0.9'>\n"

  for _, page in ipairs(pages) do
    if not page.draft then
      xml = xml .. "  <url><loc>" .. base .. page.url .. "</loc></url>\n"
    end
  end

  xml = xml .. "</urlset>"
  ctx.write_file("sitemap.xml", xml)
  ctx.log("Generated sitemap.xml with " .. tostring(#pages) .. " entries")
end

Native C-ABI plugins

For performance-critical work (image processing, custom hash strategies, binary encoders), you can write a native plugin that loads via dlopen (Linux/macOS) or LoadLibrary (Windows).

ABI contract

// nift_plugin.h — the stable C interface
struct NiftPluginInfo {
  const char *name;
  const char *version;
  const char *description;
};

struct NiftPluginApi {
  int  (*init)(void *context);
  int  (*on_file)(void *context, const char *path, char **output);
  void (*shutdown)(void);
};

// Required exports
struct NiftPluginInfo nift_plugin_info(void);
struct NiftPluginApi  nift_plugin_api(void);

Building a native plugin

# Compile as a shared library
g++ -shared -fPIC -O2 -o myplugin.so myplugin.cpp
# or on Windows:
cl /LD /O2 myplugin.cpp /Fe:myplugin.dll

# Install
cp myplugin.so my-project/plugins/
# or via CLI:
nift plugin install ./myplugin.so
