Getting started

Quickstart

Get a Nift v2 site running in under 60 seconds.

1. Install

# Linux / macOS
curl -LO https://github.com/ronggothelast/nsm-v2/releases/latest/download/nift-linux-x86_64.tar.gz
tar xzf nift-linux-x86_64.tar.gz
sudo mv nift /usr/local/bin/

See Install for all platforms.

2. Create a project

nift init my-blog
cd my-blog

3. Build

nift build
# Built 1 · cached 0 · failed 0 in 0 ms

Output lands in public/. Re-runs use the BLAKE3 incremental cache — unchanged sources are skipped.

4. Develop

nift serve
# Serving public/ at http://127.0.0.1:8080
# Watching content/ for changes

Edits to content/ trigger a rebuild and livereload. The page reloads automatically.

my-blog/
├── nift.json              # Project config (v2 schema)
├── content/               # Source files (.html, .md, .nift)
│   ├── index.html
│   ├── about.html
│   └── blog/
│       ├── first-post.html
│       └── second-post.html

│   ├── default.html
│   └── blog-post.html
├── partials/              # Reusable fragments
│   ├── header.html
│   └── footer.html
├── public/                # Build output (gitignore this)
└── plugins/               # Lua / native plugins
DirectoryPurpose
content/Your source files. Each file becomes a page. Supports HTML, Markdown, and templated files.
templates/Layout templates that wrap content. Referenced via @template(name) in content files.
partials/Reusable fragments (headers, footers, navbars). Included via @include(partial).
public/Build output. This is what gets deployed. Should be in .gitignore.
plugins/Lua scripts or native .so/.dll plugins.

Template language

Template language overview

Nift v2 uses a directive-based template language. Directives start with @, variables start with $. Everything else is literal text. This means you can write normal HTML without escaping angle brackets or braces. 

<!-- content/index.html -->
@template("default")

@set(title, "Home")
@set(description, "Welcome to my site")

<h1>Hello, $title!</h1>
<p>This page was built by Nift v2.</p>

Variables

Variables are set with @set(name, value) and referenced with $name.

Simple variables

@set(greeting, "Hello, world!")
<p>$greeting</p>
<!-- output: <p>Hello, world!</p> -->

Bracket syntax for adjacent text

@set(lang, "en")
<div class="${lang}-page">
<!-- output: <div class="en-page"> -->

Built-in variables

VariableDescriptionExample
$__file__Current source file pathcontent/blog/post.html
$__line__Current line number42
$__date__Build date (ISO 8601)2026-01-15
$__timestamp__Build timestamp (Unix)1737000000

Directives

Directives are the control flow of Nift templates. They start with @ and can take parenthesized arguments.

@template — wrap content in a layout

@template("default")
<h1>My Page</h1>

The template file at templates/default.html wraps this content. Use @content() in the template to insert the body.

@include — insert a partial

@include("header.html")
<main>...</main>
@include("footer.html")

@if / @elseif / @else — conditional

@set(loggedIn, true)

@if($loggedIn) {
  <p>Welcome back!</p>
} @else {
  <p>Please log in.</p>
}

@for — loop over a range

@for(i; 0; 5) {
  <li>Item $i</li>
}
<!-- output: <li>Item 0</li> ... <li>Item 4</li> -->

@while — loop while condition is true

@set(count, 3)
@while($count) {
  <p>Count: $count</p>
  @set(count, $count - 1)
}

@def — define a reusable block

@def(card, title, body) {
  <div class="card">
    <h3>$title</h3>
    <p>$body</p>
  </div>
}

@card("Hello", "This is a card.")

@set — set a variable

@set(name, "Nift")
@set(count, 42)
@set(active, true)
@set(total, $count + 8)  // expressions supported

@write — write to a file

@write("output/data.json") {
  {"generated": true, "count": 42}
}

@input — read a file

@input("partials/nav.html")
// Inserts the file contents at this point

@content — insert template body

Used inside template files to mark where the content goes:

<!-- templates/default.html -->
<!DOCTYPE html>
<html>
<head><title>$title</title></head>
<body>
  @include("nav.html")
  @content()
  @include("footer.html")
</body>
</html>

@exit — stop processing

@if($draft) {
  @exit()
}
// This content is skipped if $draft is true

Expressions

Expressions can be used inside directive arguments and support arithmetic, comparison, and logical operators.

OperatorDescriptionExample
+ - * /Arithmetic$a + $b
== !=Equality$x == 0
< > <= >=Comparison$count > 5
&& || !Logical$a && $b
()Grouping($a + $b) * 2

Escaping

Use \@ to output a literal @ and \\ for a literal backslash.

Email: user\@example.com     → user@example.com
Path: C:\\Users\\me           → C:\Users\me
Literal: \@notADirective()   → @notADirective()

Comments

Line comments start with // and are stripped from output:

// This comment will not appear in the output
<p>Visible content</p>
Content

Layouts and partials

Templates (layouts) wrap your content. A content file declares its template with @template("name") at the top. The template file uses @content() to insert the page body.

Partials are reusable fragments included with @include("name"). Common uses: headers, footers, navigation, sidebars.

Template inheritance

content/blog/post.html:
@template("blog-post")
@set(title, "My First Post")
<article>
  <h1>$title</h1>
  <p>Post content here...</p>
</article>

templates/blog-post.html:
@template("default")
<div class="blog-layout">
  @content()
</div>

templates/default.html:
<!DOCTYPE html>
<html>
<head><title>$title</title></head>
<body>@content()</body>
</html>

Front matter

Nift v2 supports JSON front matter at the top of content files. Values are available as variables during build. 

---
"title": "My Post",
"date": "2026-01-15",
"draft": false,
"tags": ["cpp", "ssg"]
---
@template("blog-post")
<h1>$title</h1>

Static assets

Files in content/ that are not HTML or templated (e.g., .css, .js, .png, .jpg, .svg, .pdf) are copied to the output directory as-is, preserving their relative paths. 

content/
├── index.html          → public/index.html (processed)
├── style.css           → public/style.css (copied)
├── logo.svg            → public/logo.svg (copied)
└── blog/
    └── post.html       → public/blog/post.html (processed)
Advanced

Build cache

Nift v2 uses BLAKE3 to hash every input file (content, templates, partials). On subsequent builds, unchanged files are skipped entirely. The cache is persisted as JSON in .nift/cache.json.

ScenarioBehavior
No cache file existsFull build — all files processed
1 content file changedOnly that file rebuilt; templates with matching hashes skipped
Template changedAll pages consuming that template rebuilt
Partial changedAll pages including that partial rebuilt
nift cleanDeletes output dir and cache

SIMD scanner

Template scanning uses xsimd to find @ and $ markers in 32-byte batches (AVX2). Measured 2.4× faster than scalar scanning on a 1 MiB sparse template. Falls back to scalar on non-x86 platforms.

Migration from v1

Existing Nift v1 projects (nsm.config format) can be migrated in one command: 

nift migrate /path/to/v1-project
# detected   nsm.config (v1 schema)
# written    nift.json
# preserved  nsm.config.backup
# ok         28 routes, 4 plugins, 0 warnings

What gets migrated

v1 Keyv2 KeyNotes
site_info/contentcontent_dirDirectory path preserved
site_info/outputoutput_dirDirectory path preserved
site_info/templatetemplate_dirDirectory path preserved
site_info/partialpartial_dirDirectory path preserved
site_info/site_urlbase_urlURL validated
tracked.json.nift/tracked.jsonMoved to hidden dir

Note Template syntax is backward compatible. @{ and @{end} blocks from v1 are auto-converted to @name{ } syntax.
