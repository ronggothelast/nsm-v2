Plugins

Two ways to extend the build.

Lua for ergonomics, native C ABI for performance. Same registration model: drop a file in plugins/ and Nift discovers it on startup.

Lua plugins

Each .lua file in plugins/ runs once at startup inside a sandboxed sol2 state. The nift table is the only way to talk back to the runtime. 

-- plugins/uppercase.lua
nift.directive("upper", function(args)
  return string.upper(args[1] or "")
end)

nift.directive("repeat", function(args)
  local s = args[1] or ""
  local n = tonumber(args[2]) or 1
  return string.rep(s, n)
end)

Available APIs

APIDescription
nift.directive(name, fn)Register a directive callable as @name(args...). fn receives a table of strings and returns a string.
nift.set(key, value)Set a global variable visible to all templates as $key.
nift.json.parse(str)Parse a JSON string into a Lua table. The only sandboxed file-IO equivalent.
nift.read_data(path)Read a file from data/ only — the runtime checks the path is inside the project's data/ directory.
nift.log(level, msg)Emit a structured log line. level is one of debug, info, warn, error.

What is stripped from the sandbox

  • os — no os.execute, no os.getenv
  • io — no io.open, no io.popen
  • package / require — no module loading
  • debug — no introspection escape hatches
  • dofile / loadfile / load / loadstring
  • collectgarbage — the runtime owns the GC

Memory and CPU are also bounded: a custom allocator caps each plugin's heap at 8 MiB by default, and an instruction-budget hook traps runaway loops. Both limits are configurable per-project.

Native plugins (C ABI)

For real performance — image transcoding, custom hash strategies — ship a shared library that exposes the C ABI declared in nift/plugin/abi.h. The header is committed in the repo and is the contract: it does not break across minor versions. 

// my_plugin.cpp
#include <nift/plugin/abi.h>
#include <cstring>

extern "C" const char* upper(const char* input) {
  static thread_local std::string out;
  out.assign(input);
  for (auto& c : out) c = toupper(static_cast<unsigned char>(c));
  return out.c_str();
}

extern "C" NIFT_PLUGIN_EXPORT void
nift_plugin_register(NiftPluginContext* ctx) {
  ctx->register_directive(ctx, "upper", &upper);
}

extern "C" NIFT_PLUGIN_EXPORT uint32_t
nift_plugin_abi_version(void) { return NIFT_PLUGIN_ABI_VERSION; }

Build it as a shared library and drop the result in plugins/:

clang++ -std=c++20 -fPIC -shared my_plugin.cpp -o plugins/upper.so

ABI versioning

Every plugin must export nift_plugin_abi_version and return the value of NIFT_PLUGIN_ABI_VERSION from the header it was built against. If the runtime sees a different version, it refuses to load the plugin and logs the mismatch. ABI breaks bump the major version number; minor versions stay backward-compatible.

Where to put plugins

  • plugins/*.lua — Lua, sandboxed
  • plugins/*.so / *.dylib / *.dll — native
  • plugins/disabled/ — anything in here is ignored

Order is deterministic: alphabetical by filename, with native plugins loaded first so Lua plugins can call directives they register.

Distributing a plugin

Lua plugins ship as a single file. Native plugins ship as a shared library plus, optionally, a tiny plugin.toml declaring the expected ABI version and any project-side configuration. There is no package registry — recommend installation via git submodule or direct download.
