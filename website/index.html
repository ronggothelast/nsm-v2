A static site framework written like real software.

Nift v2 is a ground-up C++20 rewrite of the original Nift static site manager. Nine modular crates, work-stealing build pipeline, SIMD-scanned templates, sandboxed Lua plugins, and a hot-reload dev server. No Node, no toolchain sprawl.

Why another generator

It is a compiler, not a script.

Most static site generators are written in scripting languages and treat speed as an afterthought. Nift v2 treats build time as a compiler problem: a parser, a typed AST, an evaluator, a cache layer, and a parallel build pipeline. You get C++ binary performance with a vocabulary that is closer to make than to a templating engine.

01 / Parser

Context-aware lexer + recursive descent

Templates are tokenized with a lexer that respects directive scope, then parsed into a typed AST. Errors point at lines and columns, not regex matches.

02 / Build

Work-stealing executor with SIMD scan

File reads use mmap. Template scans use xsimd to find directive markers in 32-byte batches. The executor steals work between cores so partial trees never block.

03 / Cache

BLAKE3-keyed incremental rebuilds

Inputs are hashed with BLAKE3, persisted as JSON, and replayed on subsequent builds. Touching a layout invalidates only the pages that consume it.

04 / Plugins

Sandboxed Lua via sol2

User scripts run with a memory cap, an instruction-budget hook, and zero filesystem or shell access. os.execute, io.open, require are stripped before user code runs.

05 / Server

Built-in dev server with file watching

nift serve binds cpp-httplib, polls your project for changes, rebuilds incrementally, and reloads. No Vite, no esbuild, no bundler config.

06 / ABI

C-ABI native plugins

For work that does not belong in Lua, ship a native plugin via dlopen/LoadLibrary against a stable C ABI. Image processing, custom hash strategies, encoders.

Drop-in

Migrate a v1 project in one command.

Existing Nift v1 projects ship with an nsm.config file and a familiar tree of content, layouts, and partials. The compat crate reads the v1 config, rewrites it to v2 schema, and validates the result. No data loss; the original file stays as a sibling.

Migration guide
