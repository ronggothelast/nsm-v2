Architecture

Nine crates, one binary.

Nift v2 is organised as a layered set of CMake targets — internally we call them crates, in the Rust sense. Each owns a single concern, depends only on its layer or below, and is independently testable. The full layout is documented in docs/adr/.

Crate layout

CrateResponsibilityKey deps
coreResult, Path, filesystem, string utils, datetimefmt, spdlog
parserLexer, AST, recursive-descent parser, evaluatorcore
runtimeLua via sol2, expression evaluator, BLAKE3 + JSON cacheparser, sol2, blake3
projectProject model: nift.config, content/layouts/partials discoverycore
buildWork-stealing pool, mmap reader, xsimd scanner, pipelineruntime, project, mio, xsimd
servercpp-httplib HTTP server, file watcher, asset minifierbuild, cpp-httplib
pluginC ABI plugin loader (dlopen/LoadLibrary)core
compatv1 nsm.config → v2 nift.config migratorproject
cliargparse, subcommands, exit-code mappingbuild, server, compat, plugin

The dependency graph is a strict DAG. core has no internal deps; cli sits at the top and is the only crate the binary target links against directly.

Build pipeline

discover  →  scan     →  parse  →  hash      →  evaluate  →  write
project       SIMD,xsimd     AST       BLAKE3       Lua sandbox    atomic
discovery     32-byte        typed    + JSON       + directives   rename(2)
              batches                  cache

The pipeline is parallel across pages but ordered within a page. A page only ever progresses left-to-right.

Work-stealing executor

Each worker thread owns a LIFO local deque. New tasks are pushed onto the local deque (LIFO improves cache locality). When a worker runs out of local work, it picks a random victim and steals from the bottom of that worker's deque (FIFO, the oldest task — most likely cache-cold for the victim, cache-warm for the thief). Exceptions raised inside a task are captured into the task's std::future and re-thrown when the pipeline calls get().

SIMD scanner

The scanner is a tight loop over xsimd 32-byte batches looking for directive markers (@, $, {, }). It dispatches to AVX2, SSE4.2, or NEON depending on the detected ISA. The tail loop handles the final <32 bytes scalar to avoid any out-of-bounds reads.

On a 1 MiB sparse-marker buffer we measure 0.37 ns/byte SIMD vs 0.89 ns/byte scalar — 2.4× speedup. The bench is checked in as tests/bench/bench_simd_scanner.cpp and runs in CI.

Cache

The cache is a JSON file at .nift/cache.json. Each entry keys a content path to:

  • BLAKE3 hash of the input
  • BLAKE3 hashes of every layout, partial, and data file the page consumed
  • The output path written
  • A monotonic build epoch

On a subsequent build, the pipeline rehashes the input and its dependency closure; a match means the previous output is still valid and the page is skipped. Touching a layout invalidates only the pages that consumed it.

JSON over CBOR was a deliberate choice — see ADR 0005. Universally inspectable beats slightly smaller for a developer-facing sidecar.

Plugin model

Lua plugins run in sol2 states with stripped libraries, a custom allocator, and an instruction-budget hook. Native plugins are shared libraries discovered via dlopen/LoadLibrary and gated on a 32-bit ABI version check. Both register against the same directive table, so a template author cannot tell — and does not need to — whether @upper is implemented in Lua or C.

Threading model

  • The executor is a fixed-size pool of std::thread::hardware_concurrency() workers.
  • The cache is read once per build and written once at the end. Workers see an immutable snapshot.
  • The Lua state is per-worker. No plugin shares state across threads.
  • The dev server runs the build pipeline on a single dedicated thread separate from the HTTP thread, so a slow rebuild never blocks the socket.
