Architecture

Deep dive into Nift v2's internal architecture. For contributors and advanced users who want to understand how the system works.

Crate dependency graph

Parser pipeline

Templates go through a 4-stage pipeline: Lex → Parse → AST → Evaluate.

StageInputOutputKey design
LexerRaw template textToken streamContext-aware: only @ and $ are special in text mode. Structural tokens ((, ), {, }, ,) only active inside directives.
ParserToken streamAST (shared_ptr<Node>)Recursive descent. Block directives (@if{}, @for{}) parsed as nested blocks. Multi-token lookahead for @name(params){body}.
AST—Tree of nodesTextNode, VarNode, DirectiveNode, BlockNode, CommentNode, ProgramNode. Shared ownership via shared_ptr (gcc 11 compatibility).
EvaluatorAST + contextOutput stringVisitor pattern. Handles all directives: @if, @for, @while, @def, @set, @write, @input, @content, @exit.

Build executor

nift::build::Pipeline::build(sources) orchestrates per-file work: 

for each source file:
  ┌──► Hash content (BLAKE3)
  │
  ├──► Cache check
  │      ├──► hit  ──► skip (cached)
  │      └──► miss ──► continue
  │
  ├──► mmap the source file
  │
  ├──► SIMD scan for @ and $ markers
  │
  ├──► Lex → Parse → AST → Evaluate
  │
  ├──► Write output (public/...)
  │
  └──► Update cache entry

The work-stealing executor distributes files across N threads. If one thread finishes early, it steals work from another thread's queue. This prevents partial dependency trees from blocking the build.

SIMD scanner

The SIMD scanner uses xsimd to process 32 bytes at a time (AVX2). It looks for @ (0x40) and $ (0x24) characters, returning a bitmask of positions. This is 2.4× faster than scalar scanning on a 1 MiB sparse template. 

// Pseudocode
for (i = 0; i < len; i += 32) {
  batch = xsimd::load_aligned(data + i);
  at_mask = xsimd::eq(batch, '@');    // 0x40
  var_mask = xsimd::eq(batch, '$');   // 0x24
  combined = xsimd::bitwise_or(at_mask, var_mask);
  if (xsimd::any(combined)) {
    // Found markers — record positions, switch to lexer
  }
}

Falls back to scalar on non-x86 platforms (ARM, WASM).

Build cache

The build cache uses BLAKE3 to hash every input file. Cache entries are persisted as JSON in .nift/cache.json. On rebuild, unchanged files are skipped entirely. Template changes invalidate all pages that consume them.

InputHashInvalidates
Content file changedRehash that fileOnly that page
Template changedRehash templateAll pages using that template
Partial changedRehash partialAll pages including that partial
Plugin changedRehash pluginAll pages (full rebuild)
